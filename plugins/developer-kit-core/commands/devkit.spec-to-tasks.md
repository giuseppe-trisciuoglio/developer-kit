# Specification to Tasks

Converts a functional specification into a list of executable, trackable tasks. This is the bridge between WHAT (specification) and HOW (implementation).

## Overview

This command reads a functional specification generated by `/developer-kit:devkit.brainstorm` and converts it into atomic, executable tasks.

**Input**: `docs/specs/YYYY-MM-DD--feature-name.md` (from devkit.brainstorm)
**Output**: `docs/tasks/YYYY-MM-DD--feature-name--tasks.md`

### Task Structure

Each task includes:
- **Title**: Descriptive name for the task
- **Description**: Functional description of what to implement
- **Acceptance Criteria**: Testable conditions for completion
- **Dependencies**: Other tasks that must complete first (if any)
- **Implementation Command**: Pre-filled command to execute this task

### Workflow Position

```
Idea → Functional Specification → Tasks → Implementation
              (devkit.brainstorm)   (this)   (devkit.feature-development)
```

## Usage

```bash
# Basic usage
/developer-kit:devkit.spec-to-tasks docs/specs/2026-03-07-user-auth.md

# With language specification
/developer-kit:devkit.spec-to-tasks --lang=spring docs/specs/2026-03-07-user-auth.md
/developer-kit:devkit.spec-to-tasks --lang=typescript docs/specs/2026-03-07-user-auth.md
/developer-kit:devkit.spec-to-tasks --lang=nestjs docs/specs/2026-03-07-user-auth.md
/developer-kit:devkit.spec-to-tasks --lang=react docs/specs/2026-03-07-user-auth.md
/developer-kit:devkit.spec-to-tasks --lang=python docs/specs/2026-03-07-user-auth.md
/developer-kit:devkit.spec-to-tasks --lang=general docs/specs/2026-03-07-user-auth.md
```

You are converting a functional specification into executable tasks. Follow a systematic approach: analyze requirements, identify dependencies, generate atomic tasks, and create a trackable task list.

## Core Principles

- **Atomic tasks**: Each task should be implementable in a single focused effort
- **Clear dependencies**: Explicitly state which tasks depend on others
- **Testable criteria**: Each task must have clear acceptance criteria
- **Functional focus**: Tasks describe WHAT, not HOW (the implementation detail comes later)
- **Use TodoWrite**: Track all progress throughout
- **No time estimates**: DO NOT provide or request time estimates

## Phase 1: Specification Analysis

**Goal**: Read and understand the functional specification

**Input**: $ARGUMENTS (spec file path)

**Actions**:

1. Create todo list with all phases
2. Parse $ARGUMENTS to extract:
   - `--lang` parameter (language/framework for implementation)
   - `spec-file` (path to specification)
3. Read the functional specification file
4. Verify the specification exists and is valid
5. If file not found, ask user for correct path via AskUserQuestion

---

## Phase 2: Requirement Extraction

**Goal**: Extract and organize requirements from the specification

**Actions**:

1. Analyze the specification for:
   - User stories and use cases
   - Business rules
   - Acceptance criteria
   - Integration requirements

2. Group related requirements:
   - Identify naturally atomic units of work
   - Note dependencies between requirements
   - Prioritize based on dependencies (what can be done first)

3. Present the extracted requirements structure to user for confirmation

---

## Phase 3: Task Decomposition

**Goal**: Break down requirements into atomic, executable tasks

**Actions**:

1. For each requirement group, create one or more tasks:
   - Each task should be implementable in 1-2 hours max
   - Tasks should have clear, testable completion criteria
   - Avoid tasks that span multiple user stories

2. For each task, define:
   - **Title**: Concise, descriptive name (e.g., "User login functionality")
   - **Description**: What the task covers functionally
   - **Acceptance Criteria**: 2-4 testable conditions
   - **Dependencies**: List task IDs this depends on (if any)

3. Map dependencies:
   - Identify which tasks must complete before others can start
   - Create a dependency graph mentally
   - Order tasks accordingly

4. Present task structure to user via AskUserQuestion for approval

---

## Phase 4: Task List Generation

**Goal**: Generate the task list markdown file

**Actions**:

1. Generate a unique task ID for each task (e.g., TASK-001, TASK-002)

2. Create the output file: `docs/tasks/YYYY-MM-DD--feature-name--tasks.md`

3. Format each task:

```markdown
### TASK-XXX: [Task Title]

**Description**: [Functional description of what this task covers]

**Acceptance Criteria**:
- [ ] Criterion 1
- [ ] Criterion 2
- [ ] Criterion 3

**Dependencies**: [TASK-YYY if applicable, otherwise "None"]

**Implementation Command**:
/developer-kit:devkit.feature-development --lang=[language] "Task: [Task Title]"
```

4. Add a task index at the top:

```markdown
# Task List: [Feature Name]

**Specification**: docs/specs/YYYY-MM-DD--feature-name.md
**Generated**: [current date]
**Language**: [language]

## Task Index

| Task ID | Title | Status | Dependencies |
|---------|-------|--------|--------------|
| TASK-001 | Task title | [ ] | - |
| TASK-002 | Task title | [ ] | TASK-001 |

## Tasks
```

5. Save the file

---

## Phase 5: Review and Confirmation

**Goal**: Verify the task list quality

**Actions**:

1. Present the generated task list to the user
2. Ask for confirmation via AskUserQuestion:
   - **Option A**: Task list looks good, proceed to summary
   - **Option B**: Modify specific tasks (specify which)
   - **Option C**: Regenerate with different decomposition

3. If modifications needed, return to Phase 3

---

## Phase 6: Summary

**Goal**: Document what was accomplished

**Actions**:

1. Mark all todos complete
2. Summarize:
    - **Specification Used**: Path to input specification
    - **Tasks Generated**: Number of tasks created
    - **Dependency Structure**: Brief overview of task dependencies
    - **Output File**: `docs/tasks/YYYY-MM-DD--feature-name--tasks.md`
    - **Next Step**: Execute tasks using devkit.feature-development

3. Provide example commands for implementing tasks:

```bash
# Example: Implement first task
/developer-kit:devkit.feature-development --lang=[language] "Task: [Task Title]"

# Example: Implement all tasks (run multiple times)
```

---

## Task Dependencies

When tasks have dependencies, the workflow is:

1. Implement tasks with no dependencies first
2. After completing a task, check if dependent tasks can now proceed
3. Use the task list to track progress

### Example Dependency Flow

```
TASK-001: User registration form (no dependencies)
    ↓
TASK-002: User login (depends on TASK-001 - uses registered users)
    ↓
TASK-003: Password reset (depends on TASK-001 - uses user data)
    ↓
TASK-004: Session management (depends on TASK-002 - uses login)
```

## Integration with devkit.feature-development

The task list generated by this command feeds directly into `/developer-kit:devkit.feature-development`:

```bash
# After generating tasks, implement each one:

# Option 1: Implement all tasks (sequentially)
/developer-kit:devkit.feature-development --lang=spring "Task: User registration"
# (complete, then)
/developer-kit:devkit.feature-development --lang=spring "Task: User login"
# ...

# Option 2: Implement tasks in dependency order
# Start with tasks that have no dependencies
# Progress through the dependency graph

# Option 3: Pick specific task to work on
/developer-kit:devkit.feature-development --lang=spring "Task: Password reset"
```
