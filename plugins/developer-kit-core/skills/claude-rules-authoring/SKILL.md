# Claude Rules Authoring

Standard for writing `.claude/rules/*.md` files — the per-project convention files Claude Code loads to follow a codebase's house rules. This skill provides the naming convention, frontmatter schema, body structure, and quality bar so rules are consistent, discoverable, and authoritative.

Use this when authoring rules by hand or refining rules emitted by the `learn` skill. The two are complementary: `learn` discovers and generates rules; this standard governs how they should look.

## Workflow

1. **Inspect what exists.** List `.claude/rules/` and read `CLAUDE.md`/`AGENTS.md` so a new rule does not duplicate an existing one and fits the established numbering band.
   ```bash
   ls .claude/rules/ 2>/dev/null
   ```
2. **Classify the rule** into one of three shapes — this drives both the body structure and whether frontmatter is needed:
   - **Policy / safety** (e.g. "never push to git", "use `git mv`"): short, imperative, *needs frontmatter* with `alwaysApply: true`.
   - **Convention** (e.g. TypeScript style, DTO organization): domain sections, usually *no frontmatter*.
   - **Pattern / guide** (e.g. "how to add a new API", module anatomy): overview + structure tree + worked code examples.
3. **Pick the filename** following the naming convention below.
4. **Write the file** from `assets/rule-template.md`, keeping the matching section set for the rule's shape.
5. **Cross-reference** sibling rules by filename and flag anything that no longer exists.

See `references/examples.md` for a complete worked example of each of the three shapes.

## Naming Convention

`NN[suffix]-topic.md` — a two-digit ordering prefix, an optional letter suffix, then a kebab-case topic.

- The **numeric prefix** groups rules into priority bands and controls load order. Suggested banding:
  - `00` — policies & safety (git, commits, comment discipline)
  - `01` — project structure / stack overview
  - `02`–`03` — base language rules (TypeScript, etc.)
  - `04`+ — framework specifics, then cross-cutting concerns (DTOs, auth, testing, env, docker…)
- The **letter suffix** (`00a`, `00b`, `15a`) inserts a sub-rule into an existing band without renumbering the rest.
- The **topic** is kebab-case and specific: `nestjs-controllers.md`, not `backend.md`.
- Prefer many focused files over one large file. **One concept per file.**

## Frontmatter

Optional YAML, used mainly for policy rules that must always be in context. Convention and pattern rules normally omit it and rely on a clear `# Title`.

```yaml
---
description: One-line summary of the rule
globs: "**/*"        # path glob(s) the rule applies to
alwaysApply: true    # true = always loaded; omit/false = context-triggered
priority: 00         # optional explicit ordering hint
---
```

Include only fields that add signal. A scoped rule might use `globs: "apps/api/**/*.ts"`; a universal safety rule uses `globs: "**/*"` with `alwaysApply: true`.

## Body Structure

- Open with `# Title` phrased as the rule itself (imperative or a clear statement of the convention), not a vague topic label.
- **Policy rules**: `## Rule` → `## Rationale` → `## Exceptions` / `## Workflow`.
- **Convention rules**: domain sections (`## Configuration`, `## Conventions`, `## Organization`) — no boilerplate headers needed.
- **Pattern/guide rules**: `## Overview`, a directory/structure tree, then worked code examples and a `## Procedure`.

## Quality Conventions

- **Authoritative, not aspirational.** State what the codebase *is*. When something was removed, say so explicitly (e.g. "`PERMISSIONS` no longer exists — use the role→feature matrix") so the agent does not resurrect dead patterns.
- **Reference real symbols and paths** from the repo — exact module names, import aliases, enum names. Generic advice belongs in CLAUDE.md, not a rule.
- **Explain why, not what.** A rule earns its tokens by stating a constraint, trade-off, or non-obvious pitfall — not by restating the obvious.
- **Use ✅ / ❌** to contrast correct vs incorrect usage where it sharpens the rule.
- **Cross-reference** related rules by filename ("see `10-authentication.md`") instead of repeating their content.
- **Split when a file covers more than one concept**, mirroring the `04-nestjs-*` / `08*-drizzle*` / `12*-testing*` family pattern.
